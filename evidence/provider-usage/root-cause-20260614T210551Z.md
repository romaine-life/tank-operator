# Provider Usage Unknown Root Cause

Captured: 2026-06-14T21:05:51Z

## User-visible failure

Slot 5 showed primary Claude usage as `unknown` while Claude secondary and
Codex had values:

- Screenshot: `/workspace/screenshots/9.png`
- Glimmung inspection id: `b1033365-7ab4-430c-b0e2-01b73045ffc0`

## Direct evidence

The primary Claude usage path was not missing UI parsing. It was returning no
usable source payload:

- Slot-local primary proxy called `https://api.anthropic.com/api/oauth/usage`.
- Anthropic returned repeated HTTP 429.
- The proxy returned `/usage/claude` HTTP 502 because there was no prior
  successful in-process usage sample to serve.
- Secondary proxy returned usage in the same slot.

Existing artifacts:

- `evidence/provider-usage/slot-5-proxy-usage-20260614T203455Z.txt`
- `evidence/provider-usage/claude-primary-proxy-logs-20260614T203455Z.txt`
- `evidence/provider-usage/claude-secondary-proxy-logs-20260614T203455Z.txt`

## Implementation fault

The home card treated a live provider scrape failure as equivalent to
`unknown` usage. That made usage availability depend on page-load timing,
proxy process lifetime, and upstream rate limiting.

The first cache patch was insufficient:

- `api-proxy` cache only worked after a first successful scrape in that proxy
  process.
- orchestrator cache only worked after a first successful scrape in that
  orchestrator process.
- neither was durable across restarts, rollouts, or cold starts.

## Claude Code evidence

The installed Claude Agent SDK exposes a structured usage control method:

- `/opt/claude-runner/node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts`
  documents `usage_EXPERIMENTAL_MAY_CHANGE_DO_NOT_RELY_ON_THIS_API_YET()`.
- The response includes `rate_limits_available` and `rate_limits` for
  claude.ai plan windows.
- This is the structured data behind Claude Code's `/usage` command.
- The installed Claude Code binary also contains `rate_limited_seeded` and
  `Usage endpoint is rate limited. Please try again in a moment.`, which
  matches the intended product behavior: keep seeded usage visible while a
  refresh is rate-limited.

That means Tank should not rely solely on browser-triggered live scraping of
`/api/oauth/usage`. Claude Code already has an SDK usage surface; Tank needs
to persist observations from it and from any successful proxy scrape.

## Fix direction implemented in this branch

- Added durable `provider_quota_snapshots` Postgres table keyed by
  `(session_scope, provider, window_id)`.
- `/api/provider-quotas` writes successful proxy scrape rows into the table,
  then serves the durable table.
- Live source errors remain in `errors`, but they no longer erase durable
  usage rows.
- Claude runner now samples the SDK usage control method at a bounded cadence
  and reports the snapshot through the existing authenticated internal
  runtime-config endpoint.
- Runtime-config maps Claude primary and Claude secondary sessions to separate
  quota providers before writing durable rows.
